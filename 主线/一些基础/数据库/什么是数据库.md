# 什么是数据库

人们通常用数据库这个术语来代表他们使用的数据库软件，这是不正确的，也因此产生了许多混淆。确切地说，数据库软件应称为数据库管理系统（DBMS）​。数据库是通过 DBMS 创建和操纵的容器，而具体它究竟是什么，形式如何，各种数据库都不一样。

简单来说：

- **数据库** - 存放数据的"仓库"
- **DBMS** - 管理这个仓库的"管理软件"

平时说的 MySQL、PostgreSQL、SQLite，严格来说都是 DBMS，而不是数据库本身。

## 为什么需要数据库

在没有数据库之前，数据可以存在文件里（比如 `.txt`、`.csv`）。但当数据量变大、用户变多、需求变复杂时，传统文件存放方式显的羸弱：

| 问题         | 文件存储                               | 数据库                                         |
| :----------- | :------------------------------------- | :--------------------------------------------- |
| 并发访问     | 多人同时读写一个文件？容易冲突甚至损坏 | DBMS 有锁机制和事务，能安全地处理并发           |
| 数据查找     | 想找某条记录？只能从头到尾扫一遍       | 有索引，查找速度极快                           |
| 数据一致性   | 写到一半断电了？文件可能只写了一半     | 事务保证要么全部成功，要么全部回滚             |
| 数据关系     | 数据之间有关联？文件里很难表达         | 关系型数据库天然支持表与表之间的关联           |
| 权限控制     | 谁都能打开文件看？                     | 可以精细控制谁能看、谁能改、谁能删             |

所以一句话：**数据库就是为了更可靠、更高效、更安全地管理数据而存在的。**

## 数据库的分类

数据库并不只有一种，根据数据的组织方式，主要分为以下几类：

### 关系型数据库（RDBMS）

这是最经典、最广泛使用的类型。数据以**表（Table）**的形式存储，表与表之间通过**关系（Relation）**连接。

**核心概念**：

- **表（Table）** - 类似 Excel 的工作表，每一列是一个字段，每一行是一条记录
- **行（Row）** - 一条具体的数据记录，也叫元组（Tuple）
- **列（Column）** - 一个数据字段，也叫属性（Attribute）
- **主键（Primary Key）** - 唯一标识一行数据的字段，不可重复、不可为空
- **外键（Foreign Key）** - 引用另一张表的主键，用于建立表与表之间的关系

**举个栗子**，一个简单的学生选课系统：

`students` 表：

| student_id | name   | age |
| :--------- | :----- | :-: |
| 1          | 张三   | 20  |
| 2          | 李四   | 21  |

`courses` 表：

| course_id | course_name |
| :-------- | :---------- |
| 101       | 数据库原理  |
| 102       | 操作系统    |

`enrollments` 表（学生和课程的关系）：

| student_id | course_id |
| :--------- | :-------- |
| 1          | 101       |
| 1          | 102       |
| 2          | 101       |

通过 `student_id` 和 `course_id` 这两个外键，三张表就串联起来了。想知道张三选了什么课？查 `enrollments` 表就行。

**常见的关系型数据库**：MySQL、PostgreSQL、SQLite、SQL Server、Oracle

### 非关系型数据库（NoSQL）

NoSQL = Not Only SQL，并不是"不用 SQL"的意思，而是"不仅仅是 SQL"。

这类数据库不要求固定的表结构，更灵活，适合处理海量、多变的数据。主要有以下几种：

| 类型             | 特点                           | 代表产品                |
| :--------------- | :----------------------------- | :---------------------- |
| 键值存储         | 最简单，一个 key 对应一个 value | Redis、Memcached        |
| 文档数据库       | 存储 JSON/BSON 格式的文档      | MongoDB、CouchDB        |
| 列族数据库       | 按列存储，适合大数据分析       | HBase、Cassandra        |
| 图数据库         | 专门处理复杂的关系网络         | Neo4j、ArangoDB         |

**什么时候用 NoSQL？**

- 数据结构不固定，经常变化（比如用户画像，每个人的属性都不一样）
- 需要极高的读写性能（比如缓存、实时计数）
- 数据量特别大，需要分布式存储

## SQL：和数据库对话的语言

要操作关系型数据库，就需要用到 **SQL**（Structured Query Language，结构化查询语言）。

SQL 不是某个数据库的专属语言，而是一种**标准**。几乎所有关系型数据库都支持 SQL，虽然各家会有一些方言差异，但核心语法是通用的。

SQL 语句主要分为几大类：

| 分类 | 全称                          | 用途           | 常见语句                         |
| :--: | :---------------------------- | :------------- | :------------------------------- |
| DDL  | Data Definition Language      | 定义数据结构   | `CREATE`、`ALTER`、`DROP`        |
| DML  | Data Manipulation Language    | 操作数据       | `SELECT`、`INSERT`、`UPDATE`、`DELETE` |
| DCL  | Data Control Language         | 控制权限       | `GRANT`、`REVOKE`                |
| TCL  | Transaction Control Language  | 管理事务       | `COMMIT`、`ROLLBACK`             |

**一个最简单的查询**：

```sql
-- 查询所有年龄大于 20 的学生
SELECT name, age FROM students WHERE age > 20;
```

这条语句做了什么？
1. `SELECT name, age` - 我只要 name 和 age 这两列
2. `FROM students` - 从 students 表里查
3. `WHERE age > 20` - 只要年龄大于 20 的

就像跟数据库说："嘿，把学生表里年龄超过 20 的人的名字和年龄告诉我。"

## ACID：事务的四大保障

数据库最核心的能力之一就是**事务（Transaction）**。事务是一组操作的集合，要么全部成功，要么全部失败，不会出现"做了一半"的尴尬局面。

> 经典场景：银行转账。A 给 B 转 100 元，A 的账户扣了 100，但 B 的账户没加上——这就出大问题了。事务保证这两步要么一起成功，要么一起回滚。

事务需要满足 **ACID** 四个特性：

- **A - 原子性（Atomicity）** - 事务是最小单位，不可再分。要么全做，要么全不做
- **C - 一致性（Consistency）** - 事务执行前后，数据库必须从一个一致状态到另一个一致状态
- **I - 隔离性（Isolation）** - 多个事务并发执行时，彼此互不干扰
- **D - 持久性（Durability）** - 事务一旦提交，结果就是永久的，即使系统崩溃也不丢失

## 数据库的基本架构

一个典型的 DBMS 内部大致长这样：

```
┌─────────────────────────────────────┐
│           应用程序 / 用户            │
│         （发送 SQL 查询）            │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│          查询处理器                  │
│  解析 SQL → 优化执行计划 → 执行     │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│          存储引擎                    │
│  管理数据的读写、索引、缓存、事务   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│          磁盘 / 文件系统             │
│      （数据最终存储的地方）          │
└─────────────────────────────────────┘
```

**你写的每一条 SQL，都会经历这样一个旅程**：从文本解析 → 语法检查 → 查询优化 → 执行引擎 → 读写磁盘 → 返回结果。

## 总结

| 概念   | 一句话解释                                      |
| :----- | :---------------------------------------------- |
| 数据库 | 有组织地存储数据的容器                          |
| DBMS   | 管理数据库的软件（MySQL、PostgreSQL 等）        |
| SQL    | 和关系型数据库对话的标准语言                    |
| 表     | 关系型数据库中存储数据的基本单位                |
| 主键   | 唯一标识一行数据的字段                          |
| 外键   | 连接不同表之间关系的桥梁                        |
| 事务   | 一组要么全成功、要么全失败的操作                |
| ACID   | 事务必须满足的四大特性                          |
| NoSQL  | 非关系型数据库，灵活应对多变数据                |
