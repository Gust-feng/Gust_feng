> 虽然 SQL 多用于数据查询，但在此之前了解数据是如何存储是很有必要的。

## 库（Database）

在使用数据库之前，首先要有一个"库"。库是什么？可以把它理解成一个**文件夹**——你的所有表、视图、索引等数据库对象，都放在这个文件夹里。

一个 DBMS 可以同时管理很多个库。比如你可能有一个 `school` 库管理学校数据，一个 `blog` 库管理博客数据，它们互不干扰。

```sql
-- 创建一个叫 school 的库
CREATE DATABASE school;
```

这条语句做的事情很简单：告诉 DBMS "帮我新建一个叫 school 的容器"。在 MySQL 里，执行完这条语句后，你甚至能在磁盘上找到一个对应的目录。

但光创建还不够，你还需要告诉 DBMS："我接下来的操作都是针对 school 这个库的。"

```sql
-- 切换到 school 库
USE school;
```

这就像在文件管理器里双击打开了 school 文件夹——之后你创建的表、插入的数据，都归这个库管。

## 表（Table）

库有了，接下来需要**表**。

如果说库是文件夹，那么表就是文件夹里的一个个**Excel 表格**。每张表存储一类数据，比如 `students` 表存学生信息，`courses` 表存课程信息。

一张表由**行（Row）**和**列（Column）**组成：

- **列**定义了数据的**结构**——有哪些字段，每个字段是什么类型
- **行**是一条条**具体的数据记录**

```
students 表：

  列 →    id    |  name  |  age  |  birthday   |  score
         -------+--------+-------+-------------+--------
  行 →     1    |  张三  |   20  | 2004-03-01  |  85.50
  行 →     2    |  李四  |   19  | 2005-06-15  |  92.00
```

创建表的 SQL：

```sql
CREATE TABLE students (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(20) NOT NULL,
    age INT,
    birthday DATE,
    score DECIMAL(5,2)
);
```

这条语句一下子涉及了三个重要概念：**数据类型**、**约束**、**主键**。别急，一个一个来。

## 数据类型：每列能存什么

为什么需要数据类型？因为计算机底层存储的全是 0 和 1，它需要知道**怎么解释**这些 0 和 1。

同样的二进制序列 `01000001`：
- 如果当作整数，它是 `65`
- 如果当作字符，它是 `A`

所以你声明 `age INT` 就是在告诉数据库："age 这一列存的是整数，请用整数的方式来存储和解读它。"

### 整数类型

| 类型       | 字节 | 范围                                    | 适合场景       |
| :--------- | :--: | :-------------------------------------- | :------------- |
| `TINYINT`  |  1   | -128 ~ 127（无符号 0 ~ 255）            | 状态码、开关   |
| `SMALLINT` |  2   | -32768 ~ 32767                         | 小范围计数     |
| `INT`      |  4   | 约 ±21 亿                              | 大多数整数场景 |
| `BIGINT`   |  8   | 约 ±922 亿亿                           | 超大 ID、时间戳 |

为什么不全用 `BIGINT`？因为**浪费空间**。一个 `TINYINT` 只占 1 个字节，`BIGINT` 占 8 个。当你有几百万行数据时，一列就差了好几 MB，在磁盘和内存上都不划算。

> **无符号（UNSIGNED）**：如果你确定一个字段不会有负数（比如年龄），可以加 `UNSIGNED`，这样 `TINYINT UNSIGNED` 的范围就从 -128~127 变成了 0~255，正数空间翻倍。

### 字符串类型

| 类型         | 说明                              | 最大长度    |
| :----------- | :-------------------------------- | :---------- |
| `CHAR(N)`    | **定长**字符串，总是占 N 个字符   | 255 字符    |
| `VARCHAR(N)` | **变长**字符串，按实际长度存储    | 65535 字节  |
| `TEXT`       | 大文本                            | 65535 字节  |
| `LONGTEXT`   | 超大文本                          | 约 4GB      |

`CHAR` 和 `VARCHAR` 的区别很关键：

```
CHAR(10) 存储 "abc"：
  [ a ][ b ][ c ][   ][   ][   ][   ][   ][   ][   ]
  ← 无论存多少，始终占 10 个字符的空间 →

VARCHAR(10) 存储 "abc"：
  [3][ a ][ b ][ c ]
  ↑ 额外 1~2 字节记录实际长度
  ← 只占实际需要的空间 →
```

**什么时候用 `CHAR`？** 当数据长度几乎固定时，比如身份证号（18 位）、手机号（11 位）、MD5 值（32 位）。因为定长意味着不需要额外存长度信息，读写效率稍高。

**什么时候用 `VARCHAR`？** 大多数时候。用户名、地址、备注这些长度变化大的字段，用 `VARCHAR` 能节省大量空间。

### 小数类型

| 类型           | 说明                 | 精度              |
| :------------- | :------------------- | :---------------- |
| `FLOAT`        | 单精度浮点数，4 字节 | 约 7 位有效数字   |
| `DOUBLE`       | 双精度浮点数，8 字节 | 约 15 位有效数字  |
| `DECIMAL(M,D)` | 精确小数             | 完全精确          |

`FLOAT` 和 `DOUBLE` 使用浮点数编码（IEEE 754），运算速度快但**有精度损失**：

```
-- 浮点数的经典陷阱
SELECT 0.1 + 0.2;
-- 结果可能是 0.30000000000000004，而不是 0.3
```

这在做数学计算时可以接受，但在涉及**金额**的场景绝对不行——你不会希望用户账户里的钱莫名其妙多了或少了 0.000001 元。

**`DECIMAL(M,D)` 是精确小数**：
- `M` 是总位数（包含小数点前后）
- `D` 是小数位数
- 比如 `DECIMAL(5,2)` 能存 -999.99 ~ 999.99

> **经验法则**：涉及钱，用 `DECIMAL`；其他科学计算、近似值，用 `FLOAT` 或 `DOUBLE`。

### 日期和时间类型

| 类型        | 格式                  | 范围                        | 说明               |
| :---------- | :-------------------- | :-------------------------- | :----------------- |
| `DATE`      | `YYYY-MM-DD`          | 1000-01-01 ~ 9999-12-31     | 只存日期           |
| `TIME`      | `HH:MM:SS`            | -838:59:59 ~ 838:59:59      | 只存时间           |
| `DATETIME`  | `YYYY-MM-DD HH:MM:SS` | 1000-01-01 ~ 9999-12-31     | 日期 + 时间        |
| `TIMESTAMP` | `YYYY-MM-DD HH:MM:SS` | 1970-01-01 ~ 2038-01-19     | 日期 + 时间 + 时区 |

`DATETIME` 和 `TIMESTAMP` 看起来很像，区别在哪？

- **`DATETIME`** 存的是你给的字面值。你存 `2024-03-01 12:00:00`，不管服务器在哪个时区，读出来就是 `2024-03-01 12:00:00`
- **`TIMESTAMP`** 会转换成 UTC 时间戳存储，读取时再按当前时区转换回来。同一条数据，在东八区读出来是 `12:00`，在零时区读出来就是 `04:00`

> 如果你的应用可能涉及多时区（比如全球用户），用 `TIMESTAMP`。如果只在本地用，`DATETIME` 更直观。
>
> 另外注意 `TIMESTAMP` 有 2038 年问题——它底层是 32 位整数存秒数，到 2038-01-19 就溢出了。不过现在很多数据库已经在逐步解决这个问题。

## NULL：一个特殊的存在

在数据库里，`NULL` 不是 0，不是空字符串 `""`，不是 false。**`NULL` 表示"未知"或"不存在"**。

这个区别很重要：

- 成绩为 `0` → 考了 0 分（确实考了，只是没得分）
- 成绩为 `NULL` → 压根没参加考试（成绩未知）

`NULL` 有一个让初学者困惑的特性——**任何与 `NULL` 的比较运算结果都是 `NULL`**，而不是 true 或 false：

```sql
-- 以下判断结果都不是 true，也不是 false，而是 NULL
SELECT NULL = NULL;   -- NULL（不是 true！）
SELECT NULL != NULL;  -- NULL（也不是 true！）
SELECT NULL > 0;      -- NULL
```

所以查找 NULL 值不能用 `=`，要用专门的语法：

```sql
-- ✗ 错误写法
SELECT * FROM students WHERE score = NULL;

-- ✓ 正确写法
SELECT * FROM students WHERE score IS NULL;
SELECT * FROM students WHERE score IS NOT NULL;
```

## 约束（Constraint）：给数据立规矩

数据类型解决了"存什么格式"的问题，**约束**解决的是"什么样的数据才允许存进来"的问题。

没有约束的表就像没有门禁的小区——什么数据都能往里塞，脏数据混进来了，后面查询和业务逻辑全乱套。

### NOT NULL —— 不许为空

```sql
name VARCHAR(20) NOT NULL
```

表示 `name` 这个字段必须有值，插入数据时不能跳过它，也不能给 `NULL`。

想想也合理——一个学生连名字都没有，这条记录还有什么意义？

### DEFAULT —— 默认值

```sql
age INT DEFAULT 18
```

如果插入数据时没有给 `age` 赋值，自动填充 `18`。

适合那些"大部分情况下值都一样"的字段，比如状态（默认"启用"）、注册时间（默认当前时间）。

### UNIQUE —— 唯一性

```sql
email VARCHAR(50) UNIQUE
```

保证这一列的每个值都不重复。两个学生不能用同一个邮箱注册。

> `UNIQUE` 允许多个 `NULL`（在大多数数据库中），因为 `NULL` 表示"未知"，两个"未知"之间不构成重复。

### CHECK —— 自定义规则

```sql
score DECIMAL(5,2) CHECK (score >= 0 AND score <= 100)
```

自己定义一条规则，不满足条件的数据直接拒绝。成绩不在 0~100 之间？不让存。

### PRIMARY KEY —— 主键

主键是最重要的约束，需要专门来聊。

在一张表里，你怎么唯一确定某一行？假设有两个叫"张三"的学生，年龄也一样，你怎么区分他们？

这就是**主键**的作用——给每一行一个**唯一标识**。

```sql
id INT PRIMARY KEY
```

主键的特点：
- **唯一**（UNIQUE）—— 不会有两行的主键值相同
- **非空**（NOT NULL）—— 每一行都必须有主键值
- **一张表只能有一个主键**

> 主键本质上就是 `NOT NULL` + `UNIQUE` 的组合，但它有更深层的含义：它是这张表里每条数据的"身份证号"。

### AUTO_INCREMENT —— 自动编号

```sql
id INT PRIMARY KEY AUTO_INCREMENT
```

有了主键，还有一个问题：每次插入数据都要自己想一个不重复的 id？太麻烦了。

`AUTO_INCREMENT` 让数据库自动帮你编号：第一条是 1，第二条是 2，第三条是 3……你完全不用操心。

```sql
-- 不需要手动给 id
INSERT INTO students (name, age) VALUES ('张三', 20);  -- id 自动为 1
INSERT INTO students (name, age) VALUES ('李四', 19);  -- id 自动为 2
```

> 注意：如果你删除了 id=2 的数据再插入新数据，新数据的 id 不会是 2，而是 3。AUTO_INCREMENT 只增不减，跳过已经用过的数字。这是为了避免 id 被复用导致混乱。

### FOREIGN KEY —— 外键

外键用来表达**表与表之间的关系**。

假设我们有一张 `classes` 表存班级信息：

```sql
CREATE TABLE classes (
    class_id INT PRIMARY KEY AUTO_INCREMENT,
    class_name VARCHAR(20) NOT NULL
);
```

然后 `students` 表里有个 `class_id` 字段，表示这个学生属于哪个班：

```sql
CREATE TABLE students (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(20) NOT NULL,
    class_id INT,
    FOREIGN KEY (class_id) REFERENCES classes(class_id)
);
```

`FOREIGN KEY (class_id) REFERENCES classes(class_id)` 这句话的意思是：

> "`students` 表的 `class_id` 必须是 `classes` 表的 `class_id` 中已经存在的值。"

如果你试图插入一个 `class_id = 999` 的学生，而 `classes` 表里没有 999 这个班，数据库会直接拒绝。这就保证了**数据的引用完整性**——不会出现"学生属于一个不存在的班级"这种荒谬情况。

## 完整例子：建一个学生管理系统

把前面的知识点串起来：

```sql
-- 1. 创建并切换到数据库
CREATE DATABASE school;
USE school;

-- 2. 创建班级表
CREATE TABLE classes (
    class_id INT PRIMARY KEY AUTO_INCREMENT,
    class_name VARCHAR(20) NOT NULL
);

-- 3. 创建学生表（引用班级表）
CREATE TABLE students (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(20) NOT NULL,
    age INT DEFAULT 18,
    birthday DATE,
    email VARCHAR(50) UNIQUE,
    score DECIMAL(5,2) CHECK (score >= 0 AND score <= 100),
    class_id INT,
    FOREIGN KEY (class_id) REFERENCES classes(class_id)
);

-- 4. 先插入班级（因为学生表引用了班级表，所以班级要先有数据）
INSERT INTO classes (class_name) VALUES ('计算机一班'), ('计算机二班');

-- 5. 插入学生
INSERT INTO students (name, age, birthday, email, score, class_id) 
VALUES ('张三', 20, '2004-03-01', 'zhangsan@school.com', 85.5, 1),
       ('李四', 19, '2005-06-15', 'lisi@school.com', 92.0, 1),
       ('王五', 21, '2003-09-20', 'wangwu@school.com', 78.0, 2);
```

## 数据操作：增删改

表和数据都有了，来看看怎么操作它们。

### INSERT —— 插入数据

```sql
-- 指定列名插入（推荐写法，清晰明了）
INSERT INTO students (name, age, score, class_id) 
VALUES ('赵六', 20, 88.5, 2);

-- 一次插入多行
INSERT INTO students (name, age, score, class_id) 
VALUES ('钱七', 22, 95.0, 1),
       ('孙八', 20, 77.5, 2);
```

> 为什么推荐指定列名？因为不指定的话，必须按表定义的列顺序给出**所有列**的值。一旦表结构变了（比如加了一列），原来的 INSERT 语句就会出错。指定列名则不会受影响。

### UPDATE —— 修改数据

```sql
-- 把张三的成绩改成 90
UPDATE students SET score = 90 WHERE name = '张三';

-- 同时修改多个字段
UPDATE students SET age = 21, score = 95.5 WHERE id = 1;

-- 给所有计算机二班的学生加 5 分
UPDATE students SET score = score + 5 WHERE class_id = 2;
```

> ⚠️ **永远不要忘了 `WHERE`！**
>
> `UPDATE students SET score = 0;` —— 没有 WHERE 条件，所有人的成绩都变成 0 了。
>
> 好的习惯是：在执行 UPDATE 之前，先用相同的 WHERE 条件执行一次 SELECT，确认影响的是哪些行。

### DELETE —— 删除数据

```sql
-- 删除特定记录
DELETE FROM students WHERE name = '赵六';

-- 删除所有成绩低于 60 的学生记录
DELETE FROM students WHERE score < 60;
```

> ⚠️ 和 UPDATE 一样，**`DELETE` 没有 `WHERE` = 清空全表**。
>
> 如果真的要清空表，有两种方式：
>
> ```sql
> -- 方式一：DELETE，逐行删除，可以回滚，但慢
> DELETE FROM students;
>
> -- 方式二：TRUNCATE，直接重建表结构，更快，但不可回滚
> TRUNCATE TABLE students;
> ```

## ALTER —— 修改表结构

表已经上线在用了，但需求变了怎么办？`ALTER TABLE` 可以在不删表重建的前提下修改表结构。

```sql
-- 添加一列
ALTER TABLE students ADD phone VARCHAR(15);

-- 删除一列（这一列的所有数据也会一起消失）
ALTER TABLE students DROP COLUMN phone;

-- 修改列的数据类型（比如 age 从 INT 改成 SMALLINT）
ALTER TABLE students MODIFY age SMALLINT;

-- 重命名列
ALTER TABLE students CHANGE birthday birth_date DATE;

-- 重命名表
ALTER TABLE students RENAME TO student_info;
```

> 在生产环境中，`ALTER TABLE` 要特别小心。对大表做结构修改可能需要很长时间，期间表可能会被锁定，影响线上服务。

## DROP —— 删除表或库

```sql
-- 删除表（表和数据一起没了）
DROP TABLE students;

-- 删除数据库（整个库和里面的所有表一起没了）
DROP DATABASE school;

-- 安全写法：先检查是否存在，避免报错
DROP TABLE IF EXISTS students;
DROP DATABASE IF EXISTS school;
```

> 经典名言："从删库到跑路。" 所以在生产环境，**永远要有备份，永远要谨慎**。

## 小结

这篇从数据存储的角度，走了一遍完整流程：

```
建库 → 建表 → 定义数据类型 → 设置约束 → 插入 / 修改 / 删除数据 → 修改表结构
```

核心知识点回顾：

| 概念         | 一句话说明                                     |
| :----------- | :--------------------------------------------- |
| 库           | 存放表的容器，可以理解成文件夹                 |
| 表           | 存储一类数据的结构，由行和列组成               |
| 数据类型     | 告诉数据库这一列的数据该怎么存、怎么解读       |
| NULL         | 表示"未知"，不是 0 也不是空字符串              |
| 约束         | 给数据设定规则，保证数据的正确性和完整性       |
| 主键         | 一行数据的"身份证号"，唯一且非空              |
| AUTO_INCREMENT | 自动递增编号，不用手动分配 id                |
| 外键         | 连接两张表的桥梁，保证引用完整性               |
| INSERT       | 往表里添加新数据                               |
| UPDATE       | 修改已有数据（别忘了 WHERE）                   |
| DELETE       | 删除数据（别忘了 WHERE）                       |
| ALTER        | 修改表结构                                     |
| DROP         | 彻底删除表或库                                 |
